Servicio para almacenar y gestionar tuplas en C:
<key, value1, value2, value3>
key: 
Cadena de caracteres de máximo 255 caracteres (sin contar el carácter nulo final).
value1: 
Cadena de caracteres de máximo 255 caracteres (sin contar el carácter nulo final).
value2: 
Vector de números float de tamaño N, donde:
N debe estar comprendido entre 1 y 32.
Se proporciona tanto N como el vector.
value3: estructura definida como: Paquete (claves.h)

El servicio ofrece una API definida en claves.h con las siguientes funciones (No se pueden modificar sus prototipos):

int destroy(void);
int set_value(char *key, char *value1, int N_value2, float *V_value2, struct Paquete value3);
int get_value(char *key, char *value1, int *N_value2, float *V_value2, struct Paquete *value3);
int modify_value(char *key, char *value1, int N_value2, float *V_value2, struct Paquete value3);
int delete_key(char *key);
int exist(char *key);
Todas las operaciones deben realizarse de forma atómica.

A - Versión no distribuida

    Se debe implementar una versión no distribuida del servicio siguiendo una arquitectura monolítica.

    Requisitos:

    - Implementar en claves.c la lógica

    - La estructura de datos es libre (lista, árbol, etc.), pero:
    No debe imponer límite en el número de elementos.
    No puede requerir software adicional (por ejemplo, bases de datos externas).
    Versión inicial: Lista enlazada (Simplicidad y paralelismo)
    Opciones adicionales: Árbol binario
    Tabla de hashes, Versión óptima, valorar complejidad a futuro y paralalismo

    - Generar una biblioteca dinámica denominada libclaves.so a partir de la implementación anterior.

    - Programa cliente (cliente.c) que:

    - Utiliza exclusivamente la API definida en claves.h.

    - Se enlace dinámicamente con libclaves.so.

    - Sirva para probar el funcionamiento.

    - Elaborar un plan de pruebas completo...

B – Versión distribuida con colas de mensajes POSIX

    Se debe transformar el servicio anterior en un sistema distribuido utilizando exclusivamente colas de mensajes POSIX.
    mqueue.h

    La arquitectura debe separar:

    - Un servidor que gestione el almacenamiento.
    - Un cliente que invoque la API de manera remota.

    Requisitos:

    Implementar el servidor (servidor-mq.c):

        Recibe peticiones mediante colas de mensajes.

        Invoca las funciones de libclaves.so.

        Devuelve el resultado al cliente.

        Paralelismo.

    Implementar el proxy del lado del cliente (proxy-mq.c):

        Implementa las mismas funciones de la API.

        Envía las peticiones al servidor mediante colas POSIX.

        Espera la respuesta y la devuelve al programa cliente.

        Genera la biblioteca dinámica libproxyclaves.so.

        Utilizar el mismo cliente.c (A).

        No puede contener referencias a colas de mensajes.

        Debe enlazarse ahora con libproxyclaves.so.

    Gestionar excepciones:

    -1: Servicio (Clave inexistente, N fuera de rango, etc.).

    -2: Comunicación (Colas de mensajes).

    Si value1 supera 255 caracteres o N_value2 > 32, se devuelve -1 sin contactar con el servidor.

    Elaborar el mismo plan de pruebas que en la Parte A y ejecutarlo en la versión distribuida con colas de mensajes POSIX.

Se debe entregar:

    - cliente.c
    - claves.c
    - servidor-mq.c
    - proxy-mq.c
    - Archivos auxiliares necesarios (.c, .h)
    
    Un Makefile que genere:
        - libclaves.so
        - libproxyclaves.so
        - Ejecutable del servidor (servidor_mq)
        - Cliente sin colas (enlazado con libclaves.so)
        - Cliente con colas (enlazado con libproxyclaves.so)

Restricciones:

    No se pueden modificar los prototipos de claves.h.
    No se puede limitar el número máximo de elementos almacenados.
    No se puede utilizar software externo.
    Todas las operaciones deben ser atómicas.
    Solo se pueden usar colas de mensajes POSIX en la versión distribuida.
    El cliente no puede contener código relacionado con colas de mensajes POSIX.
